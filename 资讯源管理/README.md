# 📚 资讯源管理系统

> 集中管理 AI 资讯追踪系统的所有信息源，跟踪接入状态，持续迭代。

## 目录结构

```
资讯源管理/
├── README.md                # 本文件
├── 源清单.md                # 主清单（一表看全部源的状态）
├── 已对接/                  # ✅ 已成功对接并稳定运行的源
│   └── <板块>_<源名>.md
├── 待对接/                  # 🟡 已确认接入方式，等待实施
│   └── <板块>_<源名>.md
├── 待预研/                  # 🔵 接入方式不明确 / 内容有效性待验证
│   └── <板块>_<源名>.md
└── 已废弃/                  # ⚫ 已停止使用 / 接入失败的源
    └── <板块>_<源名>.md
```

## 状态定义

| 状态 | 标识 | 含义 | 触发条件 |
|------|------|------|---------|
| **已对接** | ✅ | 已成功接入并稳定运行 ≥ 7 天 | 抓取成功率 ≥ 95%、内容相关性验证通过 |
| **待对接** | 🟡 | 接入方式明确，待实施或正在实施 | 抓取路径已确认，源在"源清单"中标记 P0/P1 |
| **待预研** | 🔵 | 接入方式不明确，或内容有效性待验证 | 源在"源清单"中标记 P2/P3，或无官方 RSS/API |
| **已废弃** | ⚫ | 已停止使用、接入失败、或被替代 | 抓取持续失败、内容重复、内容质量低 |

## 状态流转

```
[新源发现] → 待预研 (🔵)
              ↓ 调研通过
            待对接 (🟡)
              ↓ 实施完成，运行 ≥ 7 天验证
            已对接 (✅)
              ↓ 出现持续问题
            已废弃 (⚫) ← 也可从待对接/待预研直接废弃
```

## 单个源的记录文件结构

每个源在对应状态下有自己的详情文件，格式：

```markdown
# <源名>

## 基本信息
- **板块**：🧠 LLM 发展 / 💻 编程 Agent / 🧍 数字人 / 📦 其他
- **URL**：链接
- **抓取方式**：L1 官方 RSS / L2 官方 API / L3 RSSHub / L5 网页抓取
- **更新频率**：X 篇/周

## 接入状态
- **状态**：🟡 待对接
- **优先级**：P0
- **负责人**：AI 调研专家
- **创建日期**：YYYY-MM-DD

## 接入计划
- [ ] 抓取脚本开发
- [ ] 内容质量验证
- [ ] 连续运行 7 天

## 备注
（任何关于该源的特殊说明）
```

## 主清单：源清单.md

统一表格记录所有源，支持快速筛选。详见 [源清单.md](源清单.md)

## 工作流

### 新增一个源
1. 在 `源清单.md` 添加一行（状态：待预研）
2. 在 `待预研/` 创建详情文件
3. 调研完成后移到 `待对接/`
4. 实施完成后移到 `已对接/`
5. 提交：`./scripts/news-commit.sh "feat(资讯源): 新增 XXX 源"`

### 更新源状态
1. 编辑 `源清单.md` 更新状态列
2. 移动详情文件到对应目录
3. 提交
